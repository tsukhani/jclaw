**Docs Drift Workflow**

Sweep JClaw's hand-maintained documentation and realign it with what the code actually does — versions, framework levels, command names, file paths, feature/endpoint coverage. Triage drift before editing, fix the legitimate items surgically, and stop at a local commit. Use `/usr/bin/git` for every git invocation (project convention).

**Arguments**

`$ARGUMENTS` may be one of:

- *(empty)* → full sweep of every **Tier-1** doc, plus a **Tier-2** detect-and-report pass.
- a file path or cluster name (e.g. `AGENTS.md`, `README`, `docs/user-guide/`, `docs/user-guide/agents.md`, `sidecar`) → scope the run to that file or cluster only. Still report Tier-2 drift for awareness.
- `check` (or `--check` / `dry-run`) → **detect-and-report only**: produce the full drift table and make **no edits and no commit**.

Reject anything else with a clear message; do not guess.

---

**Scope — three tiers, fixed**

- **Tier 1 — auto-update** (edit directly, surgically):
  - `README.md`
  - `CLAUDE.md`
  - `AGENTS.md` (the Codex peer of CLAUDE.md — keep shared facts in sync, preserve its distinct addressee/voice)
  - `docs/user-guide/*.md` (all files)
  - `sidecar/flux/README.md`
- **Tier 2 — detect & report only** (never hand-rewrite):
  - `docs/architecture/*.md` — generated by `bmad-document-project`. Collect drift into the report and recommend regenerating via `/bmad-document-project`; do **not** edit these files.
- **Out of scope — never touch**:
  - `openspec/**` (tool-managed proposal/spec lifecycle; `archive/` is frozen history)
  - `skills/**/SKILL.md` and skill references (skill/tool definitions, mostly vendored)
  - `workspace/main/*.md` (`SOUL.md`, `IDENTITY.md`, `AGENT.md`, `BOOTSTRAP.md`, `USER.md` — runtime agent persona / Standing Orders, not codebase docs)
  - `.claude/commands/*.md` (the slash commands themselves)

**Phase 1 — Establish ground truth (read code before touching any doc)**

Build a small "truth sheet" of the facts docs most often get wrong, sourced from the canonical files — never from another doc:

1. `conf/application.conf` → `application.version` (the live version) and any status line.
2. `frontend/package.json` → Nuxt major version, the `packageManager` pnpm pin (and its `+sha512` hash), Node `engines`, and the `scripts` block (the real `pnpm` targets: `dev`, `build`, `lint`, `test`, `typecheck`, `stylelint`, …).
3. `build.gradle.kts` + `settings.gradle.kts` → JDK level, the Play 1.x fork wiring, and notable dependency choices (e.g. the OkHttp-5 outbound stack, Lucene version).
4. `jclaw.sh` → the real subcommands (`setup`, `init-worktree`, `start`, `stop`, `restart`, `status`, `test`, `dist`, `bundle`, `loadtest`, …) and the **`play autotest`** convention (`play test` is interactive — never document it as the CI command).
5. `conf/routes` + `app/controllers/` → the API surface under `/api/**` that contracts/feature docs reference.
6. `.githooks/` → actual `pre-commit` (lint-staged on staged frontend files) and `pre-push` (full suite, per-HEAD cache) behavior.
7. `frontend/nuxt.config.ts`, `frontend/app/pages/`, `frontend/app/composables/` → the frontend feature surface the user guide describes (channels, chat, agents, subagents, tasks, reminders, settings, logs/dashboard, skills/tools/MCP).

Treat **CLAUDE.md as a strong but fallible source**: it is usually the convention of record, but it can drift too — verify its claims against the code rather than assuming it is correct, and never propagate a CLAUDE.md fact into another doc without confirming it in the codebase.

**Phase 2 — Detect drift per doc**

For each in-scope file, cross-check its claims against the truth sheet. Look for:

- **Stale version / status** strings (e.g. `v0.1.0-alpha`, an old `0.x.y`).
- **Wrong framework versions** (e.g. `Nuxt 3` where the code is on Nuxt 4; outdated JDK or Play levels).
- **Renamed / removed commands** (e.g. `play test` / `play auto-test` → `play autotest`; pnpm script names; `./jclaw.sh` subcommands that changed).
- **Broken file / directory references** — verify every path a doc names still exists.
- **Endpoint / feature mismatches** — feature lists or `/api` endpoints that no longer match `conf/routes`, controllers, or frontend pages.
- **Coverage gaps** — shipped capabilities with no doc coverage (e.g. a new channel or feature with no user-guide page). Flag these; do not silently auto-author whole new pages.
- **Broken internal markdown links** between docs.

Assign each drift item a verdict: **FIX** (Tier-1 file) or **REPORT** (Tier-2 architecture doc). Group by file.

**Phase 3 — Present the drift table & confirm**

Present a table (file · drift item · `current → correct` · tier · FIX/REPORT) covering every finding, and get a quick confirmation before editing — same triage-before-mutate discipline as `/sonar`. In `check` mode, stop here: this table **is** the deliverable.

**Phase 4 — Apply Tier-1 fixes (surgical)**

7. Edit only the drifted lines, via the Edit tool. Preserve each doc's voice, heading structure, and formatting. Do **not** rewrite whole sections, restyle prose that isn't drifted, or add speculative content. Every changed line must trace to a confirmed drift item.
8. Keep `CLAUDE.md` and `AGENTS.md` consistent on **shared** facts (versions, commands, framework levels) while preserving each file's distinct addressee ("Claude" vs "Codex") and any intentionally different guidance.
9. For a genuine **coverage gap** that warrants a new user-guide page or section, confirm scope with the user (in Phase 3) before authoring — don't fabricate a page unprompted.
10. **Tier 2:** make no edits to `docs/architecture/*.md`. Compile their drift into the final report with the `/bmad-document-project` regeneration recommendation.

**Phase 5 — Validate (lightweight; no test suite needed for docs)**

11. Re-verify that every command, script, subcommand, and file path the **edited** docs now reference actually exists (`./jclaw.sh` subcommands, `pnpm` scripts in `package.json`, routes/controllers, on-disk paths).
12. Confirm internal markdown links in edited files resolve to real targets.
13. No edited Tier-1 doc lives under `frontend/**`, so the frontend lint/test gates don't apply — but if that ever changes, run `cd frontend && pnpm test` for those edits.

**Phase 6 — Commit locally & report (stop here; never push)**

14. Stage only the docs you actually changed with `/usr/bin/git add <paths>`. Docs-only commits short-circuit the `pre-commit` hook (no staged `frontend/**` files), so there's zero lint cost.
15. Create a local commit:
    ```bash
    /usr/bin/git commit -m "$(cat <<'EOF'
    docs: realign <scope> with codebase (v<VERSION>)

    <one short paragraph: what drifted and what was corrected — e.g. version
    bump propagation, Nuxt 3→4, play test→autotest, fixed file references.>

    Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
    EOF
    )"
    ```
16. Report: a table of Tier-1 files with their fixed-item counts; the Tier-2 architecture-doc drift list with the `/bmad-document-project` recommendation; the commit hash; and any coverage gaps deferred for a follow-up. **Do not push** — `/deploy` is the only path that ships docs to the remotes; leave the commit for the user to review and release.

---

**Hard rules**

- Triage before mutating — present the drift table and get a quick confirmation before editing any file (skip straight to the table in `check` mode and stop).
- Stop at the **local commit**. Never `git push`, tag, or release — `/deploy` owns that.
- Use `/usr/bin/git` for every git invocation. Never `--no-verify`.
- **Surgical edits only**: change drifted facts, never rewrite, reflow, or restyle documentation that isn't drifted. Match the surrounding voice.
- **Never hand-edit `docs/architecture/*.md`** — they're a `bmad-document-project` artifact; detect & report, then recommend regeneration.
- **Never touch** `openspec/**`, `skills/**`, `workspace/main/*.md`, or `.claude/commands/*.md`.
- **Verify before asserting**: confirm a command, script, endpoint, or path exists in the code before writing it into a doc. Never copy a claim from one doc into another without checking the codebase — and treat CLAUDE.md as fallible, not gospel.
- Don't auto-author entire new pages for coverage gaps without confirming scope first.
