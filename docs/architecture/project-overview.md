# JClaw — Project Overview

> Generated by `bmad-document-project` — 2026-04-15. Source of truth: codebase at this date. Re-scan with `/bmad-document-project` if structure drifts.

## Purpose

JClaw is Abundent's **AI-powered automation platform**: a long-running Java service that hosts conversational LLM agents, schedules background tasks, executes tools (shell, filesystem, web, browser), and exposes a unified Web/Slack/Telegram/WhatsApp chat surface. It replaces two predecessor projects (`OpenClaw` Node.js, `JavaClaw` Spring Boot) with a single Java-first runtime on Play 1.x + a Nuxt 3 SPA.

Current version: **v0.7.18** (status: v0.1.0-alpha / work in progress per CLAUDE.md).

## Repository type

**Multi-part**. One git repo, two runtimes that ship as a single deploy unit. The Play dist zip is backend-only; the Nuxt SPA is generated and staged into `public/spa/` by whichever consumer deploys it (bare-metal `jclaw.sh start` or the Docker multi-stage build).

| Part | Root | Type | Runtime |
|---|---|---|---|
| `backend` | `/` (Play convention: `app/`, `conf/`) | backend | Play 1.x / Java 25 / JDK Zulu |
| `frontend` | `/frontend` | web | Nuxt 3 / Vue 3 / TypeScript / Tailwind |

## High-level architecture

```
┌──────────────────┐      ┌────────────────────────────────────────┐
│   Nuxt 3 SPA     │ ───► │  Play 1.x backend (:9000)              │
│   (SPA only,     │  /api│                                        │
│    ssr: false)   │ ◄─── │  • ApiController suite (auth-gated)    │
└──────────────────┘      │  • AgentRouter → AgentRunner pipeline  │
         ▲                │     → LlmProvider (OpenAI/Ollama/OR)   │
         │ /_nuxt static  │     → ToolRegistry execution loop      │
         └────────────────│  • Play Jobs: TaskPoller @30s,         │
                          │                 CleanupJobs, etc.      │
  Inbound webhooks  ────► │  • Channel adapters: Slack/Telegram/WA │
  (telegram/slack/wa)     │  • JPA on H2 (dev) / PostgreSQL (prod) │
                          └────────────────────────────────────────┘
```

### Core backend flows

- **Chat (web)** — `ApiChatController` → `AgentRunner.run/runStreaming` → `LlmProvider` + tool loop → persists `Message` rows → SSE to browser via `ApiEventsController`.
- **Channel inbound** — `WebhookSlackController` / `WebhookTelegramController` / `WebhookWhatsAppController` → `AgentRouter.resolve` (3-tier: peer → channel → main agent) → `AgentRunner.run` → outbound via `channels.Channel.resolve()`.
- **Tasks** — `ApiTasksController` creates rows in `task`; `TaskPollerJob @Every("30s")` claims `PENDING` tasks whose `next_run_at ≤ now`, runs each in a virtual-thread executor, handles CRON re-scheduling and exponential-backoff retries.
- **Skills** — files-on-disk skill definitions; `SkillLoader` + `SkillPromotionService` promote authored skills into the global registry. Promoted skills declare `commands:` that get persisted into `agent_skill_allowed_tool` for shell-exec allowlisting (the DB — not the workspace file — is authoritative).

### Frontend

Nuxt 3 SPA with `ssr: false`. Single global layout (`layouts/default.vue`), global auth middleware (`middleware/auth.global.ts`) that redirects to `/login` when `/api/config` 4xxs. All backend access through `$fetch` / `useApi` / `useApiMutation`. An `EventSource` connection to `/api/events` provides server-push notifications (e.g. `skill.promoted`, streaming tokens).
