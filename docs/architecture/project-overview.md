# JClaw — Project Overview

> Generated by `bmad-document-project` — 2026-06-29 (deep rescan). Source of truth: codebase at this date. Re-scan with `/bmad-document-project` if structure drifts.

## Purpose

JClaw is Abundent's **AI-powered automation platform**: a long-running Java service that hosts conversational LLM agents, schedules background tasks, executes tools (shell, filesystem, web, browser, documents, image/video generation), and exposes a unified Web/Slack/Telegram/WhatsApp chat surface. It draws feature designs from two predecessor projects (`OpenClaw` Node.js, `JavaClaw` Spring Boot) but shares no code — it's an original Java-first runtime on a customized Play 1.x fork + a Nuxt 4 SPA.

Current version: see `application.version` in `conf/application.conf` (the source of truth; status: pre-v1 / beta, work in progress per AGENTS.md).

## Repository type

**Multi-part monorepo**. One git repo, two runtimes that ship as a single deploy unit. The Play backend serves the prebuilt Nuxt SPA from `public/spa/`; the self-contained `jclaw-bundle.zip` (and the GHCR container image) bake in the framework, resolved deps, precompiled classes, and the built SPA so the only host requirement is a Java 25 JRE.

| Part | Root | Type | Runtime |
|---|---|---|---|
| `backend` | `/` (Play convention: `app/`, `conf/`) | backend | Play 1.x fork / Java 25 / JDK Zulu |
| `frontend` | `/frontend` | web | Nuxt 4 / Vue 3 / TypeScript / Tailwind v4 |

## High-level architecture

```
┌──────────────────┐      ┌────────────────────────────────────────┐
│   Nuxt 4 SPA     │ ───► │  Play 1.x backend (:9000 / :9443)      │
│   (SPA only,     │  /api│                                        │
│    ssr: false)   │ ◄─── │  • Api* controller suite (auth-gated)  │
└──────────────────┘      │  • AgentRouter → AgentRunner pipeline   │
         ▲                │     → LlmProvider (OkHttp 5, SSE)       │
         │ /_nuxt static  │     → ToolRegistry execution loop       │
         └────────────────│  • Jobs: TaskPoller (db-scheduler),     │
                          │          cleanup, probes, MCP startup   │
  Inbound webhooks  ────► │  • Channel adapters: Slack/Telegram/WA  │
  (telegram/slack/wa)     │  • JPA → H2 (dev) / PostgreSQL (prod)   │
                          │  • Lucene 10 full-text index            │
                          └────────────────────────────────────────┘
```

### Core backend flows

- **Chat (web)** — `ApiChatController` → `agents.AgentRunner` → `LlmProvider` (OkHttp 5 single-shot/SSE) + tool-call loop (`ToolCallLoopRunner`, bounded rounds) → persists `Message` rows → SSE to browser via `ApiEventsController`.
- **Channel inbound** — `WebhookSlackController` / `WebhookTelegramController` / `WebhookWhatsAppController` (and the Telegram/Slack polling/socket runners) → `agents.AgentRouter.resolve` (3-tier: peer binding → channel binding → main agent) → `AgentRunner` → outbound via the matching `channels.Channel` adapter.
- **Tasks** — `ApiTasksController` writes `task` rows; **db-scheduler** claims due tasks atomically (row-version), runs each via `TaskExecutionHandler` on virtual threads, and handles CRON re-scheduling + backoff retries. Reminders are tasks with `auto_delete_on_complete`.
- **Skills** — files-on-disk skill definitions; `SkillLoader` + `SkillPromotionService` promote authored skills into the registry. Promoted skills' blessed shell commands persist into `agent_skill_allowed_tool` (the DB — not the workspace file — is authoritative for shell-exec allowlisting).
- **Search** — `Message`, `Task`, `TaskRunMessage`, `Memory`, and `SubagentRun` rows sync to per-scope Lucene indexes via JPA `@PostPersist`/`@PostUpdate`/`@PostRemove` hooks; queries run through `DirectLuceneMessageSearchRepository` with NRT `maybeRefresh`.

### Frontend

Nuxt 4 SPA with `ssr: false`. Single global layout (`layouts/default.vue`), global auth middleware (`middleware/auth.global.ts`) that routes to `/login` (or `/setup-password` on first boot) based on auth + password-set state. Backend access goes through `$fetch`, `useApiParsed` (Zod-validated reads), and `useApiMutation` (POST/PUT/DELETE). A singleton `EventSource` to `/api/events` (in `useEventBus`) provides server-push notifications and streaming tokens. State is `useState`-based singletons — no Pinia.
