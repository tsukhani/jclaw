## Synthesis â€” Clean-Code / OOP / SOLID / GRASP / Java-25

JClaw's app/ Java is in solid clean-code shape for a pre-v1 alpha: it leans hard on records, sealed result types, and Java-25 idioms, and the framework-mandated Play static-controller/active-record patterns are used correctly. The dominant smell by a wide margin is shallow copy-paste DRY â€” the same scaffold (debounce buffers, media-send tails, scanner JSON walks, transactional UPDATE wrappers, string-contract markers) re-implemented across sibling files where one missed copy silently diverges. Beyond DRY, two genuine bugs stand out: a CRLF-rewriting edit path and an unguarded NPE outside a sealed-result try/catch. Nearly everything else is cosmetic idiom/doc polish.

### Cross-cutting themes

#### `[high]` Copy-pasted scaffolding across sibling classes that drifts independently

The single most repeated pattern in the audit: a mechanism is implemented once, then hand-copied into 2-6 sibling sites that share the structural skeleton but differ only in a constant, a column name, a builder type, or a content extractor. The Telegram inbound-coalescing buffers are the worst â€” three classes duplicate the entire ConcurrentHashMap/daemon-scheduler/Bucket/debounce/Throwable-swallow scaffold, and divergence has ALREADY happened (TelegramMediaGroupBuffer lacks resetForTest and silently uses a lossy 7-arg InboundMessage ctor that drops botMentioned/messageId/replyContext). The same shape recurs in TelegramChannel's six trySend{Photo,Document,Voice,Audio,Video} tails (photo/document already drifted by inlining the log instead of calling logMediaSent), the scanner JSON-walk/collect-â‰¤3-labels loops, McpConnectionManager's three persist* UPDATE-with-swallow methods, the two Lucene-marker finders in Commands, and four near-identical Lucene backfill methods. The unifying risk is fix-asymmetry: a correctness or policy fix applied to one lane and forgotten on the others.

**Recommendation:** Treat the Telegram coalescing buffers as the one extraction worth doing now (extract an IdleDebounceBuffer<K> owning the map/scheduler/debounce/flush lifecycle, parameterized by key+merge), because divergence is already proven and the duplicated Throwable-swallow is correctness-load-bearing. For the rest, prefer small in-place helpers over framework-style abstractions: a Supplier<Message> upload tail in TelegramChannel, a collectLabels/optString pair in ConfiguredHashScanner, a persistTimestamp(serverId,column) in McpConnectionManager, a runIdLikePattern(runId) in Commands. Skip the @MappedSuperclass for the two config entities â€” Play's enhancer-bound static finders can't be hoisted, so the abstraction removes only two trivial fields. Per the project's Simplicity-First guideline, do NOT manufacture shared interfaces/base classes for the low-line-count enum and LlmProvider cases.

## Synthesis â€” Performance / Memory

JClaw's hot LLM/streaming dispatch paths are largely sound, but the request-serving and background-job layers carry a cluster of real perf defects: two genuine read-path N+1s on user-facing list endpoints, three unbounded/under-bounded static maps that leak for the JVM lifetime, a 60s call-timeout that makes large Whisper-model downloads structurally impossible, and a VT Thread.sleep backoff loop that trips the project's own documented JDK-25 ForkJoinPool starvation hazard. None are critical (no data loss, no OOM under normal load), but the high/medium tier is concrete and mostly low-effort to fix. The long tail of redundant DB round-trips and per-row write loops is real but bounded and cold-path, so it ranks low.

### Cross-cutting themes

#### `[medium]` Unbounded / under-bounded static caches leak for the JVM lifetime

Several process-wide static maps have no production eviction path. PendingTranscripts.futures retains a CompletableFuture<String> (full transcript) per audio attachment ever processed â€” a genuine slow heap leak proportional to lifetime audio volume. JpaMemoryStore's embedding cache bounds entry COUNT (maximumSize=10_000) but keys on the full unbounded Memory.text (@Column TEXT), so the documented ~60MB heap bound is provably false and large key strings stay pinned for the 24h expireAfterWrite window even after source rows are deleted. TelegramStreamingSink.LAST_NOTIFIER_FIRE_MS and TelegramPollingRunner's REBUILD_COUNT/LAST_REBUILD_AT grow with distinct conversations/tokens and are never evicted on unregister â€” bounded to a smaller failure/rebuild population, so lower risk.

**Recommendation:** Add remove-on-consume to PendingTranscripts (covering the failure-sentinel path) and a content-hash key (record EmbeddingKey(String model, long textHash)) to JpaMemoryStore so the documented heap bound holds. For the Telegram maps, evict on unregister and add a cheap opportunistic removeIf sweep keyed on the cooldown window. These are the highest-leverage fixes because they convert lifetime-unbounded growth into bounded state with one- to few-line changes.

Affected: `app/services/transcription/PendingTranscripts.java`, `app/memory/JpaMemoryStore.java`, `app/channels/TelegramStreamingSink.java`, `app/channels/TelegramPollingRunner.java`

#### `[high]` Read-path N+1 / wide-scan on user-facing list endpoints

List endpoints enrich each page row with a per-row query or a table-wide scan. conversationToMap fires SessionCompaction.count (a real per-call COUNT, since SessionCompaction extends Model) once per conversation across a page of up to 100 rows â€” a textbook N+1, up to ~100 extra COUNTs per list call. ApiSubagentRuns.list pulls a min-500-row, ever-growing slice of the append-only EventLog (ordered timestamp DESC) and full-JSON-parses every row to extract two fields, discarding all rows not on the current page â€” O(min(500,history)) work to enrich O(page) rows, and the 500 floor crowds the page out once history exceeds it.

**Recommendation:** For conversations, mirror the proven bulk-GROUP-BY pass already in ApiTasksController.list: one SELECT ... COUNT(sc) ... WHERE conversation.id IN :ids GROUP BY ..., thread the precomputed count in (default absent to 0). For subagent runs, persist mode as a column on SubagentRun (its durable home), or at minimum bound the scan to the page's run_ids instead of a blanket category fetch. Both have an established in-repo pattern to copy.

Affected: `app/controllers/ApiConversationsController.java`, `app/controllers/ApiSubagentRunsController.java`

#### `[high]` Missing per-call deadline override on streaming downloads through the 60s general client

The large-model GET uses the bare HttpFactories.general() client whose GENERAL_CLIENT sets callTimeout(60s). OkHttp's callTimeout bounds the ENTIRE call including response-body read, so a 514MBâ€“1031MB (Medium/Large) model stream cannot complete in 60s on any normal link â€” OkHttp aborts mid-download, deletes the .part file, and surfaces a confusing IOException. Large-model local transcription is effectively undownloadable; the symptom looks like a random failure at 60 seconds. A second, conditional concern (ensureAvailable's future having no overall deadline) only becomes relevant AFTER this call deadline is removed.

**Recommendation:** Override the per-call deadline on the streaming GET via Call.timeout().clearTimeout() / Call.timeout(Duration.ZERO), bounding the read with readTimeout only â€” exactly the callTimeout(0) pattern llmStreaming() uses and the HttpFactories Javadoc explicitly endorses (no new client needed). Keep the 60s budget on the HEAD. Once that lands, add a generous future.orTimeout ceiling on ensureAvailable to bound the post-fix slow-drip stall.

Affected: `app/services/transcription/WhisperModelManager.java`

#### `[medium]` Blocking / sleeping on threads that shouldn't block â€” VT and inline-publisher hazards

Two distinct blocking-on-hot-path issues. TaskExecutionHandler.findTaskWithRaceBackoff does a bare Thread.sleep(20ms)Ă—4 inside the db-scheduler .execute() lambda, which runs on a virtual thread; a burst of newly-scheduled tasks firing near-simultaneously (the exact INSERT/poll race this guards) is precisely the many-concurrent-VTs-in-Thread.sleep scenario the project's JDK-8373224 constraint warns causes ForkJoinPool work-queue starvation â€” and the same package's LoadTestHarness was already redesigned to avoid it. NotificationBus.publish loops future.get(3s) one-at-a-time over a 2-thread pool, so N stalled SSE listeners serialize and block the inline publisher (skill promotion, task lifecycle) for ~ceil(N/2)Ă—3s, defeating its own 'a slow writer can never block fan-out' invariant. EventLogger (adjacent durability gap) has no time-based flush and flush() early-returns when shuttingDown, so a clean shutdown discards the pending tail and scheduled-task-only deployments lag the events DB until BATCH_SIZE trips.

**Recommendation:** TaskExecutionHandler: replace the busy-sleep retry with a scheduler-driven re-fire (re-schedule the task_instance a few tens of ms out and return), or run the backoff on a shared platform-thread ScheduledExecutorService like LoadTestHarness already does. NotificationBus: size the executor to listener count (cached/VT pool) so the per-listener 3s cap actually holds, or compute one shared deadline and get() each future against the remaining time. EventLogger: add an @Every(30s) flush job and flush once in markShuttingDown() before flipping the file-only flag.

Affected: `app/services/TaskExecutionHandler.java`, `app/services/NotificationBus.java`, `app/services/EventLogger.java`

#### `[medium]` Per-iteration / quadratic CPU work that scales with payload or table size

Work that grows super-linearly with input within its (bounded) cap. #### `[low]` VT pinning via synchronized blocks around blocking I/O

Two real instances of the project's flagged hazard â€” a synchronized monitor held across blocking I/O on a virtual thread pins the carrier. McpStdioTransport.send() is synchronized around BufferedWriter write/flush to a subprocess pipe; PlaywrightBrowserTool.ensureBrowserInstalled holds synchronized(PlaywrightBrowserTool.class) across proc.waitFor() for a (potentially minutes-long) Chromium download. Both are genuine pinning patterns, but blast radius is narrow: stdio send is mutually exclusive by design (one in-flight writer per connection, pipe drains instantly in the common case), and the browser install is strictly once-per-JVM, short-circuited by a volatile fast path and chromiumPreinstalled() on any real deployment.

**Recommendation:** Swap the intrinsic locks for ReentrantLock (which releases the carrier while blocked and is already imported in PlaywrightBrowserTool for BrowserSession), or serialize stdio writes through a single-writer queue. Low priority given the bounded blast radius, but the fix is mechanical and preserves the mutual-exclusion intent.

Affected: `app/mcp/transport/McpStdioTransport.java`, `app/tools/PlaywrightBrowserTool.java`
